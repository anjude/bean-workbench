# AGENTS.md

## 命名原则

Agent 指南文件和相关文档应使用中性、通用的名称。

优先使用以下文件名：

- `AGENTS.md`
- `.agents/skills/*`

避免使用带有平台标识的名称，除非该文件本身就是为某个平台定制的。

## 这样做的原因

- 让工作台在不同工具和 Agent 运行时之间保持可移植性。
- 让这些说明更像项目文档，而不是某个产品的品牌附件。
- 当同一个仓库被不同自动化入口使用时，减少不必要的改动。

## 实践规则

如果文件是为了指导 Agent 行为，就应该按它在工作台里的职责命名，而不是按读取它的工具命名。

## Skill 命名规则

工作台 `.agents/skills/` 下的 skill 目录名统一使用中性、可扩展、可分组的 kebab-case 命名，默认格式如下：

```text
{领域前缀}-{可选子域}-{能力后缀}
```

命名约束：

1. `领域前缀` 必填，用于表达 skill 主要服务的业务域或技术域。
2. `可选子域` 按需添加，用于进一步区分具体仓、品牌、平台或对象。
3. `能力后缀` 必填，用于表达这个 skill 解决什么动作或流程，如 `api-flow`、`dev-flow`、`review-flow`、`knowledge-base`。
4. 统一使用小写英文和连字符，不使用空格、下划线、拼音缩写混搭或品牌式命名。
5. 避免直接使用工具品牌、Agent 品牌或个人化命名；除非该 skill 明确只服务某个平台适配层。

推荐前缀：

- `backend-*`：后端、数据库、领域模型、接口、测试、wire 等后端能力
- `frontend-*`：Web/H5/Nuxt 等前端能力
- `uni-*`：uni-app 相关能力
- `content-*`：内容生产、内容复盘、知识整理、选题等内容能力
- `workbench-*`：工作台路由、任务、上下文编排、自进化等工作台能力
- `carbon-*`：carbon 品牌特化能力

实践规则：

- 新增 skill 时，优先先定领域前缀，再定能力后缀，避免先想一个“好记但不可归类”的名字。
- 同一领域下的 skill 命名要保持平行结构，例如 `backend-superone-api-flow`、`backend-superone-dev-flow`、`backend-superone-test-review-flow`。
- 历史 skill 如果暂时未迁移，可保留旧名继续兼容；新增和重构时应逐步收敛到统一规则。
- 业务仓内沉淀出的可复用 skill，稳定后应迁移到工作台 `.agents/skills/`，并移除带有平台或个人品牌色彩的目录命名。

## 工作台规则

1. 工作台内的所有 `md` 文档，包括 `skill` 相关文档，统一使用中文；专业术语和业务术语可以保留英文。
2. 除非是明确的业务需求，否则不应该直接改动业务仓内容。
3. 工作台用于沉淀个人 `skill`、知识库、任务记录和脚本，并承担业务仓之间的全局路由、上下文聚合和跨仓编排职责。
4. 完成业务需求后，如果出现可复用经验、反复问题或意图识别失败，应反向优化工作台的 `skill`、知识库或脚本。
5. 新增或接入业务子仓时，必须先做工作台级整体检查：同步更新 `business-repo/` 索引、路由规则、相关 `skill`、任务登记和知识库入口；必要时补充专属工作流或沉淀记录，不能只加子模块不改工作台引用。
6. 新建任务文件夹前，先检查最近的任务目录和 `task/registry.md`；如果任务属于同类问题、零碎子任务或同一上下文链条，优先合并到已有任务目录继续共用上下文，避免拆成多个孤立任务。

## 目录职责

- `.agents/skills/`：工作台维护的通用 Agent skill 源目录。
- `docs/`：工作流、模板、架构规范、目标管理等工作台辅助内容。
- `knowledge-base/`：业务仓、系统和公共知识沉淀。
- `task/`：工作台任务记录、热任务索引和上下文登记。
- `archive/`：工作台历史任务、过期规则和阶段性材料归档。
- `script/`：工作台常用脚本。
- `business-repo/`：业务仓集合，包含后端、前端、协议等独立仓；只在明确业务需求时改动。

## 自进化触发条件

满足任一条件时，应考虑更新工作台沉淀：

1. Agent 连续两轮以上没有识别清楚用户意图。
2. 同一路径的操作、排错或验证问题多次复现。
3. 某个业务仓需求暴露出稳定可复用的流程、约束或知识。
4. 热任务需要更快路由到固定业务仓、文档或上下文。

更新优先级：先补 `task/registry.md` 或痛点记录，再沉淀到 `knowledge-base/`，最后在稳定后升级为 `skill` 或脚本。
