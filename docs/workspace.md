# 工作台规范

## 目标

工作台用于沉淀个人 Agent 能力，包括 `skill`、知识库、任务索引和常用脚本，并承担业务仓之间的全局路由、上下文聚合和跨仓编排职责。

## 核心闭环

1. 用户提出需求。
2. 工作台根据热任务、知识库和路由 skill 定位业务仓、协议仓与上下文。
3. Agent 在对应业务仓完成代码或文档产出；跨仓影响由工作台编排。
4. 需求结束后复盘可复用经验、反复问题和上下文缺口。
5. 将稳定经验沉淀回 `skill`、`knowledge-base/` 或 `script/`。

## 目录职责

- `.agents/skills/`：工作台内维护的 skill 源目录。
- `docs/`：工作流、模板、架构规范、目标管理等工作台辅助内容。
- `knowledge-base/`：可复用知识库，按公共知识、业务仓和系统分层。
- `task/`：工作台任务记录和热任务索引。
- `archive/`：工作台历史任务、过期规则和阶段性材料归档。
- `script/`：工作台辅助脚本。
- `business-repo/`：业务仓集合，包含后端、前端、协议等独立仓，不承载工作台规则。

## 变更边界

1. 工作台建设默认只改根级工作台目录。
2. 除非是明确业务需求，不直接改动 `business-repo/` 下的业务仓内容。
3. 工作台文档不复制业务仓的大段实现细节，只记录路由、约束、知识索引和可复用经验。
4. 第三方原始文档、许可证和临时导入代码默认保持原样。
5. 新增业务子仓时，工作台层必须联动检查 `business-repo/` 索引、`task/registry.md`、`knowledge-base/`、相关 `skill` 和工作流文档，确保路由、职责和沉淀入口一致。
6. 新建任务目录前，必须先检查最近几个任务目录；若属于同类、零碎补充或同一上下文链条，优先合并到现有任务继续记录。

## 文档原则

1. 工作台 `md` 文档统一使用中文。
2. 专业术语、业务术语、代码标识、接口路径可以保留英文。
3. 根目录保持轻量，详细规范放入 `docs/`。
4. 避免重复内容：入口文档只做导航，细节放到专门文档。

## docs 分区

- `workflows/`：工作台常用工作流。
- `templates/`：任务、复盘、知识沉淀等模板。
- `architecture/`：工作台目录结构、边界和演进规范。
- `goals/`：工作台目标、阶段规划和验收标准。
