# 任务管理规范

## 目标

`task/` 用于记录工作台建设任务、热任务、冷任务和上下文范围，帮助 Agent 快速理解最近在做什么。

## 文件职责

- `task/registry.md`：热任务、冷任务、任务目录和资源范围索引。
- `task/YYMMDD_slug/`：单个工作台任务目录，例如 `260614_workbench-init/`。
- `task/YYMMDD_slug/README.md`：任务说明、边界、状态和沉淀记录入口。

## 边界

工作台任务只记录工作台层面的目标、规则、沉淀和路由信息。业务仓具体实现、接口变更、测试细节应记录在业务仓自己的文档或提交中。

## 任务命名

使用 `YYMMDD_slug/` 目录格式。`slug` 使用短英文或拼音，保持可搜索。

## 任务目录建议

- `README.md`：任务目标、范围、边界、状态。
- `notes.md`：过程记录和上下文摘录，按需创建。
- `review.md`：任务完成后的复盘和可沉淀内容，按需创建。

## 热任务规则

热任务用于加速路由到最近上下文。任务完成或长期不活跃后，应从热任务移到冷任务。

## 子仓接入任务

当工作台接入新的业务子仓时，应创建或更新一个对应任务目录，记录：

1. 子仓名称与仓位
2. 需要同步的 `skill`
3. 需要更新的知识库和路由提示
4. 是否需要新增工作流或检查脚本
5. 验证结果和遗留风险

这类任务用于保证“加子仓”不是只改 `.gitmodules`，而是完成工作台级整体收口。

## 提交流程

工作台根目录应提供统一的 `make commit` 命令，用于提交当前工作台全部变更。文档和任务记录中提到“提交工作台内容”时，默认指向该命令，而不是在对话里手工拼命令。提交信息直接取 `make commit` 后面的参数，多个参数用空格拼接；不带参数时使用默认提交信息。
