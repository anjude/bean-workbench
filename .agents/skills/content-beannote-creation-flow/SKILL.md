# 内容创作全流程

## 触发后先做三件事

每次使用本 skill，先执行以下步骤，不要直接写正文：

1. 读取 `business-repo/beannote/docs/content-playbook.md`
   - 继承 beannote 当前账号定位、文风偏好、最新用户反馈。
   - 项目级 playbook 优先级高于本 skill 的旧规则。

2. 读取目标一级目录的 `README.md`
   - 操作 `01_in_production/` 前，必须先读 `01_in_production/README.md`。
   - 如 README 与本 skill 冲突，以 README 为准。

3. 判断内容平台
   - 用户明确说公众号：按公众号文章生产。
   - 用户明确说小红书：按小红书图文生产。
   - 未说明平台：按中性内容生产，两边都能改。

4. 读取 `references/创作准则.md` 并逐条对照
   - 必须逐条检查是否违反禁用写法（自问自答、设问引导、二元对立、排比升华、定义式开头、完美过渡等）
   - 必须确认开篇人设（"大家好，这里是豆哥的笔记📒"）
   - 未读取 `references/创作准则.md` 并逐条对照前，不得开始写正文。

5. 读取 `references/正文质检清单.md`
   - 必须在写作前知道最终要交付哪些可验证检查项。
   - 不允许只凭印象说"已检查"、"符合要求"。
   - 最终 `发布检查清单.md` 必须逐项记录通过/需修改/不适用和简短证据。

## 默认落盘规则

只要用户是在“创作一篇内容”，默认需要落盘到 `01_in_production/`。

### 目录命名

目录必须使用：

```text
YYYY-MM-DD_主题简述
```

示例：

```text
2026-05-10_段永平泡泡玛特恋爱梗
```

### 默认文件策略

不要为了流程完整而制造文件。默认采用轻量产物，按内容复杂度加文件。

#### 公众号文章默认生成

| 文件 | 用途 |
|---|---|
| `公众号文章.md` | 可直接发布的完整稿。文件开头含选题判断（写给谁、一句话结论、核心问题、标题草案），正文紧跟其后 |
| `发布检查清单.md` | 合规、文风、事实源、自检结果 |
| `复盘.md` | 只记录生产过程、用户反馈和规则沉淀 |

#### 小红书图文默认生成

| 文件 | 用途 |
|---|---|
| `图文脚本.md` | 按自然页数拆分的图文脚本。文件开头含选题判断，正文紧跟其后 |
| `纯享正文.md` | 可直接复制发布的干净正文 |
| `发布检查清单.md` | 合规、文风、事实源、自检结果 |
| `复盘.md` | 只记录生产过程、用户反馈和规则沉淀 |

#### 按需生成

| 文件 | 何时需要 |
|---|---|
| `图文脚本.md` | 公众号文章要拆成小红书、用户明确要求图文脚本 |
| `纯享正文.md` | 用户需要跨平台复制版，或从图文脚本提取正文 |

原则：少而够用。不要因为模板存在就强行生成。

## 一次成稿流程

### 0. 自进化判断

每次创作、修改、发布检查、用户反馈后，都要主动判断：这次问题是当前稿的一次性问题，还是可复用规则缺口。

必须主动优化规则的情况：

- 用户明确指出某种写法不要再出现。
- 同类问题在不同稿件中重复出现。
- 发布检查发现某条禁用写法没有被现有清单覆盖。
- 当前 skill、memory、references 之间规则冲突或表达含糊。
- 生成内容时临时想出一条有效判断标准，且以后明显能复用。

不需要更新规则的情况：

- 只是当前选题的事实修正、标题取舍、局部措辞偏好。
- 用户只要求本篇换角度，不代表长期偏好变化。
- 修改会让规则过窄，影响其他类型内容正常创作。

主动优化顺序：

1. 先改当前稿，保证用户眼前问题解决。
2. 再判断规则应落在哪里：
   - 长期账号偏好、用户稳定反馈：更新 `business-repo/beannote/docs/content-playbook.md`
   - 本 skill 的流程动作、交付标准、自进化机制：更新 `SKILL.md`
   - 具体写作禁令、句式禁区、文风判断：更新 `references/创作准则.md`
   - 可验证的交付检查项：更新 `references/正文质检清单.md`
   - 模板结构或固定栏目：更新 `references/模板合集.md`
3. 最后更新当前主题 `复盘.md`，记录本次反馈和规则沉淀。

写入规则时要克制：

- 写成可执行判断，不写空泛口号。
- 附 1-3 个禁用例子或改法，方便下次检查。
- 不把一次性偏好升级成全局规则。
- 不重复写多处；如必须多处同步，分清 memory、skill、reference 的职责。

### 1. 选题判断

每次写正文前必须先完成选题判断，直接写在正文文件开头（公众号文章.md 或图文脚本.md 的选题区），明确：

- 这篇写给谁
- 一句话结论
- 只讲清楚的 1 个核心问题
- 3 个标题草案
- 参考资料和事实风险

选题判断可详可略。简单选题几行就够，复杂选题再展开内容路径和事实风险。不要写成空模板。

**选题排雷**：在选题判断阶段，必须检查选题是否属于烂大街模板。典型特征：

- 读者看到标题就能猜到全文结构和结论（"存到X万要注意这些坑"、"到了这个年纪一定要做这几件事"）
- 同一结构在财商号/理财号里被反复使用，换数字换马甲
- 核心逻辑是"到了某个节点→这些坑→这样处理"

如果命中以上任何一条，直接告诉用户这个选题烂大街了，建议放弃或找反常识切口。不要试图靠文风优化去救一个选题本身老化的文章。

### 2. 正文策略

写正文前先确定：

- 主判断：这篇到底想让读者多理解什么
- 氛围：这篇的情绪底色是什么。教训、追忆、讽刺、好奇、愤怒？氛围决定全文语感和结尾走向。写时代变迁用追忆比教训自然，写收割陷阱用讽刺比科普有力。
- 入口：热点梗、个人观察、认知冲突、具体场景
- 增量信息：本文比普通评论、普通科普、普通理财建议多说了什么
- 支撑物：数据、案例、场景、类比、步骤、框架或边界，至少选一种写具体
- 合规边界：只守硬红线，不主动加码
- 结尾方式：轻轻收住，或用调侃回到问题，不强行升华

不同类型的抓手：

- 理财方法：不要只给步骤，要写清楚为什么普通人会做错、这个方法解决哪个心理或行为问题。
- 认知误区：不要只说"不要这样想"，要把错误想法的诱惑写出来，再给更好的判断框架。
- 公司/行业：不要只复述新闻，要写出商业模式、财务数据、行业位置、估值锚或产业链传导中的关键一环。
- 市场事件：不要只写热闹，要写它改变了什么预期、影响谁、普通人怎么观察。
- 个人笔记：不要把"我觉得"当论据，自述只做入口，正文主体必须回到观察和判断。

去平庸标准：

- 如果正文只剩"长期主义、提升认知、值得关注、未来可期、控制风险"这类空话，必须重写。
- 如果标题拿掉后，全文可以套到任何一个选题上，必须重写。
- 如果每段只是在解释上一段为什么对，没有新事实、新观察、新判断或新方法，必须删改。

### 3. 主文生成

公众号文章：

- 生成 `公众号文章.md`
- 不默认生成 `纯享正文.md`
- 不默认生成 `图文脚本.md`
- 如果用户要跨平台，再拆成图文脚本或纯享正文
- 正文末尾追加两项（用 `---` 分隔）：

  **精彩引言**：从文中提炼 1-2 句最有冲击力的金句，加粗标记，用于转发摘要或封面引用。

  **封面 prompt**：可用 AI 生成的文章封面描述语。要求元素简洁，一个简单的画面描述，不要用明显的股票元素。风格方向：成人黑童话、心理寓言、超现实情绪插画、独立动画截图、表现主义、情绪象征、粗粝手绘、孤独感、存在主义气质。画面比例 16:9。

小红书图文：

- 生成 `图文脚本.md`
- 提取 `纯享正文.md`
- 页数按内容自然决定，通常 4-8 页，不强行凑 6 页
- 每页字数按阅读节奏决定，不为凑模板拆页

### 4. 发布检查

生成 `发布检查清单.md`，必须包含：

- 合规红线
- 文风检查
- 爆款 5 维度
- 信息密度检查（自述/观察/判断篇幅比例、每段是否有新增信息量）
- 是否有 AI 味表达
- 标题、封面、事实源发布前确认项

### 5. 复盘

生成或更新 `复盘.md`，保持简短：

- 记录本次生产中可复用的结构
- 记录用户反馈
- 记录是否触发自进化；如果触发，写明更新了哪些文件和原因
- 如果反馈暴露出规则盲区，立即修改 memory、skill 或 references，不要只改当前稿

## 公众号文章一次可发布标准

公众号文章不要写成“小红书短句合集”，也不要写成“标准评论文”。核心是：像一个真的人在认真聊这件事。

### 语气

- 可以轻松，可以调侃，但不要油。
- 允许有一点生硬转折，比如“不过，谁知道...”，这比过度顺滑更像真人。
- 不要把所有转场都磨平。过度自然，反而像 AI。
- 不要解释笑点，直接写反应。
  - 不好：这个梗好笑，是因为反差足够大。
  - 更好：哈哈哈，这反差也太大了。

### 读者位置

- 不要预设读者想错了。
- 不要用纠偏式表达：
  - 别把这个理解成...
  - 不要以为...
  - 很多人会误以为...
- 不要替读者规定重点：
  - 真正值得看的...
  - 这比讨论 X 重要得多
  - 这才是...
- 改法：顺着语境自然往下聊。

### 信息密度

每一段都要提供新东西，至少满足一个：

- 新事实
- 新观察
- 新判断
- 新例子
- 新情绪
- 新问题

删除没有新增信息量的垫话：

- 不好量化，不代表不重要。
- 这套框架当然重要。
- 它要求投资者理解的东西变多了。
- 表面看...往深一点看...
- 当然是好事。
- 这本身没问题。
- 并不是坏事。

如果一句话只是“解释前一句为什么对”，优先删除或换成具体细节。

不要用无意义的价值平衡句软化观点。如果一句话只是为了显得客观、稳妥、两边都照顾，但没有提供新观察或新判断，直接删掉。

### 结构

- 开头可以用梗，但不要一直围着梗打转。
- 中段要给出具体判断维度，不要只说“情绪价值很重要”。
- 结尾不要复述前文判断。
- 结尾可以用轻微调侃收住，也可以留一个有余味的小观察。

### 重点标注

- 可以少量使用粗体。
- 每篇 3-5 处即可。
- 只加粗真正值得停一下的句子。
- 不要满屏加粗，不要把粗体当情绪放大器。

## 账号表达写法

适用于所有公众号文章、小红书图文、选题拆解和内容改稿。

豆小匠的内容不能只是"正确"，还要有辨识度。默认写法是：先给一个能站住的判断，再用场景、数字、案例、框架把它托起来。

- 不做资料搬运：信息要被整理成判断。
- 不做温吞科普：知识点要落到真实问题。
- 不做情绪喊单：情绪要被事实、方法或边界托住。
- 不做万能鸡汤：每个结论都要有具体对象、具体场景或具体后果。

### 推荐结构

1. 人设开场或直接进入事件/问题
2. 抛出主判断或强反差
3. 展开事实、场景、案例或数据
4. 给出判断框架或观察路径
5. 说清边界，尤其是投资合规边界
6. 用轻观察、调侃或开放问题收住

### 具体判断维度要写出来

不要只写：

```text
普通人要提升认知。
```

要写成：

```text
很多人以为提升认知是多看几篇文章。
真到亏钱的时候，问题往往不是不知道道理，而是仓位、现金流和情绪承受力没对齐。
所以这篇不聊大道理，只聊一个动作：买之前先写下最坏情况。
```

### 避免重复

如果正文中已经讲过“持续做 IP 的能力”，结尾就不要再复述“到底有没有持续做 IP 的能力”。

结尾可以换成新的轻观察：

```text
不过，谁知道大道这次到底是不是在喊单呢哈哈哈

话说回来，一个头像能让投资圈聊这么久，本身就挺泡泡玛特的。
```

## 合规红线

投资理财内容，只守以下硬红线：

- 不承诺保本保收益、不写"稳赚""保本""无风险"
- 不直接喊人买/卖具体标的（"买入X""上车X""赶紧卖Y"）
- 不诱导开户或交易
- 必须保留免责声明：

```text
内容仅供参考，不构成投资建议。投资有风险，决策需谨慎。
```

红线以外的都可以写：

- 可以聊具体公司、具体产品，给出自己的观察和判断
- 可以提历史收益率、回撤数据，作为事实引用
- 可以讨论名人动作和逻辑，只是不能写成"跟着他买"
- 可以说"我觉得有意思""值得关注""我在关注"，这是观点不是指令
- 可以分析投资框架、商业模式、风险收益比

判断标准：**读者看完之后，是多了理解，还是多了冲动？** 多了理解就写，多了冲动就改。

## 交付前硬检查

每次交付前必须检查：

### 操作真实性检查

- [ ] 已实际读取 `business-repo/beannote/docs/content-playbook.md`
- [ ] 已实际读取目标目录 `README.md`
- [ ] 已实际读取 `references/创作准则.md`
- [ ] 已实际读取 `references/正文质检清单.md`
- [ ] `发布检查清单.md` 不是空模板，含具体检查证据
- [ ] 最终回复只描述已经真实完成的动作，不把计划说成结果

### 文件检查

- [ ] 目录是否是 `YYYY-MM-DD_主题简述`
- [ ] 公众号文章是否包含选题区 + 正文，外加 `发布检查清单.md`、`复盘.md`
- [ ] 小红书图文是否包含选题区 + 正文（图文脚本 + 纯享正文），外加 `发布检查清单.md`、`复盘.md`
- [ ] 额外脚本、跨平台正文是否只在需要时生成

### 正文检查

- [ ] 正文无第二人称"你"（互动问题可例外）
- [ ] 无中文破折号、英文 em dash
- [ ] 无角引号
- [ ] 无滥用引号包裹关键词
- [ ] 无保本保收益承诺、"稳赚""无风险"等绝对表述
- [ ] 已保留免责声明
- [ ] 每一段都有新增信息量
- [ ] 没有强行升华
- [ ] 没有主持人口吻强行转场
- [ ] 没有复述前文判断当结尾
- [ ] 粗体 3-5 处，不泛滥
- [ ] 正文末尾附有精彩引言和封面 prompt

### 反馈后处理

用户指出问题时：

1. 先改当前稿，不辩解
2. 判断是一次性问题还是规则盲区
3. 规则盲区要同步更新：
   - 当前主题 `复盘.md`
   - `business-repo/beannote/docs/content-playbook.md`
   - 本 skill 或 `references/` 中对应规则
4. 更新后再次检查当前稿，确认同类问题已清掉
5. 最终回复要明确说明：改了当前稿；如有规则沉淀，说明沉淀到了哪个文件
